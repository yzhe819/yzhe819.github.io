    # 1.0 传输层概述

    只有主机才拥有的层次，为应用层提供通信服务并使用网络层的服务

    # 1.1 传输层功能

    传输层的功能:

    • 传输层提供进程和进程之间的逻辑通信

      • 网络层提供了主机之间的逻辑通信

    • 复用和分用

    • 传输层对收到的报文进行差错检测

    • 传输层有两个协议（传输单元是报文段）

    # 2.1 UDP概论

    UDP只在IP数据报服务上增加了很少功能，即复用分用和差错检测功能

    UDP的主要特点：

    1. UDP是无连接的，减少开销和发送数据之前的时延

    2. UDP使用最大努力交付，即不保证可靠交付（由应用层保证可靠交付）

    3. UDP是面向报文的，适用于一次性传输少量数据的网络应用

    4. UPD没有拥塞控制，适合很多实时应用

    5. UDP首部开销小，8B，TCP20B

    # 3.1 TCP协议的特点

    1. TCP是面向连接（虚连接）的传输层协议。eg: 打电话

    2. 每一条TCP连接只能有两个端点，每一条TCP连接只能是点对点的。

    3. TCP提供可靠交付的服务，无差错，不丢失，不重复，按序到达 （可靠有序，不丢不重）

    4. TCP提供全双工通信

      • 发送缓存 - 准备发送的数据&已发送但尚未收到确认的数据
      • 接收缓存 - 按需到达但尚未被应用程序读取的数据&不按序到达的数据

    5. TCP面向字节流 - [流：流入到进程或者从进程中流出的字节序列]

      • TCP把应用程序交下来的数据看成仅仅是一连串的无结构字节流

    # 3.2 TCP报文段的首部格式

    **窗口：**指的是发送本报文段的一方的接受窗口，即现在允许对方发送的数据量

    **检验和：**检验首部+数据，检验时要加上12B伪首部，第四个字段为6

    **紧急指针：**URG=1时才有意义，指出本报文段中紧急数据的字节数

    **选项：**最大报文段长度MSS，窗口扩大，时间戳，选择确认...

    # 3.4 TCP可靠传输

    TCP是提供可靠传输，UDP这种本身还是不可靠传输的就再靠应用层解决了

    # 3.4.1 序号

    TCP根据下方数据链路层的MTU（最大传输单元）来随即将数据切割成好几端并且进行编号

    # 3.4.2 确认

    发送方每一次发送数据之后都需要接收方进行确认。

    通常确认帧使用累计确认，或者当接受方有数据要发的时候，可以把确认帧捎带上（捎带确认）

    累计确认机制，就是从第一个丢失的字节开始请求丢失的报文段。如图中456丢失，78到达，但仍然请求发送的数据序号是4

    # 3.4.3 重传

    把所有已经发送的报文段的RTT进行加权平均往返时间计算。

    收到三个对于某个特定报文段的冗余ACK后，系统默认接下来的报文段丢失，直接重传。（快速重传技术）

    # 3.5 TCP流量控制

    流量控制：让发送方慢点发送，要让接收方来得及接受

    TCP利用滑动窗口机制实现流量控制

    # 3.6 TCP拥塞控制

    流量控制：点到点问题

    拥塞控制：全局性问题

    # 3.6.1 拥塞控制的四种算法

    1. 慢开始
    2. 拥塞避免
    3. 快重传
    4. 快恢复

    # 3.6.2 慢开始和拥塞避免

    这里开始时以指数形式增长，ssthresh的意思是慢开始门限，代表从这个地方注入的报文段就比较多了，需要开始慢速增加了。

    之后一段都是线性增长，每次增加1，直至达到24的时候（网络拥塞状态） 瞬间将cwnd设置为1，同时调整原来的ssthresh的值到之前达到网络拥塞状态的1/2,（这里是24减半到12）

    重复以上步骤，但是注意此时ssthresh变了之后线性增长的转折点也变了

    # 3.6.3 快重传和快恢复

    收到三个重复的确认（冗余ACK） - >执行快重传算法

    之前都和3.6.2一样，在至达到24的时候（网络拥塞状态） 调整原来的cwnd的值到之前达到网络拥塞状态的1/2,（这里是24减半到12）然后从新的cwnd开始执行拥塞避免“加法增大”
